HtmlSocket是一个反向Ajax框架,支持主流浏览器:Chrome IE5.5+ Firefox Safari Opera

常见的反向Ajax方案及其优缺点:

  • 1、FlashSocket
      优点:FlashSocket和websocket都是基于流的方式传输数据,但FlashSocket不需要对传输的数据编解码,
      所以效率比websocket高,并且Flash可以很方便与JavaScript交互.国内的各种页游足以证明Flash很高效。
      缺点:需要安装插件;移动设备支持不好,目前已被Adobe抛弃。

  • 2、WebSocket
      优点:无需插件支持,基于流传输,编程简单,主流浏览器公司都在推,是今后的发展趋势
      缺点:规范还不稳定,老版本的浏览器不支持。

  • 3、LongPolling
      优点:前端实现简单;几乎兼容目前所有在用的浏览器
      缺点:后端需要特殊处理:挂起连接,连接状态管理较复杂

  • 4、LongConnect
      优点:基于流传输,效率可以媲美flash方式
      缺点:数据传输可能不完整,前端需要特殊处理;不是所有浏览器都支持,后端同样需要挂起连接

  • 基于定时轮询和JavaApplet的方式没多大意义,在此不予分析

透过以上数据我们可以看到,目前还有很大一部分浏览器不支持Websocket。而HtmlSocket能根据浏览器自动选择传
输方式,并且支持自动重连.相比其他类似框架,HtmlSocket拥有一套编解码流程,开发者只需要设定消息的编解码逻
辑,即可快速开发一套推送系统。后端采用java开发,连接状态管理,点对点消息转发,群组消息转发,广播消息等功能
均已封装,无需依赖其他组件,可以方便嵌入到任何Java应用,集成到应用时只需要定制自己的消息格式即可,下个版本
将支持跨语言,可以很方便和PHP Python C#等结合使用
API
客户端提供两个接口: 
         1. HtmlSockect.Settings:设置连接参数
           ==>new HtmlSockect.Settings('url', 'id','topic');
             =>url[url]:提供推送的url地址,如:http://localhost:8877/
             =>topic[string]:该客户端订阅的话题,可用于群消息,如:group
             =>id[string]:连接的ID,用于区分不同的客户端,如:userId
           ==>encode[function]:将收到的数据解码为消息格式传递给onmessage
           ==>onmessage[function]:收到消息时回调此方法   
           ==>decode[function]:发送前编码消息为传输格式
           
         2. HtmlSockect:传输对象
           ==>new HtmlSockect(setting);
             =>setting[HtmlSockect.Settings]:连接相关设置
           ==>sendTo(toId[string],message[*]):发送P2P消息
           ==>broadcastGroup(subject[string],message[*]):发送组播消息
服务端提供以下接口:
	    1. IMessage:消息接口,开发者实现该接口定义消息格式
	    2. ICodec  :消息编解码类,负责编码和解码消息
	    3. ISubscriber:消息订阅者,订阅指定主题的消息
	    4. IConnListener:连接监听器,连接建立和关闭时会回调 
	    5. WSContext:上下文,可以发送消息[P2P,组播,广播]
